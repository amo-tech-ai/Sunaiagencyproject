# Prompt: Supabase Storage for Document Uploads

> Target: Let wizard Step 1 users upload documents (pitch decks, brand guidelines, business plans) that feed into the AI analysis pipeline and persist across sessions.

---

## Current State

StepBusinessContext (C30) has a visual upload zone placeholder that accepts drag-and-drop but does not actually store files. The edge function server at index.tsx mounts wizard-routes and ai-routes on Hono. The db.tsx factory provides adminClient (service-role) and userClient (JWT-scoped). The frontend lib/supabase.ts has a getSupabaseClient singleton and the api helper for edge function calls. AuthContext provides the user's accessToken.

There is no Supabase Storage bucket created yet. No upload routes exist on the server. No file metadata is stored in wizard_sessions or wizard_answers.

## What Needs to Happen

### Server: Bucket Initialization

The edge function server should idempotently create a private bucket on startup. Name it with the required make prefix (make-283466b6-wizard-uploads). Check if the bucket already exists before creating. Private bucket means all access goes through signed URLs generated by the server.

### Server: Upload Route

Add a POST route for file uploads. The route accepts multipart form data with the file, the sessionId, and an optional label (pitch-deck, brand-guide, business-plan, other). The server should validate file size (max 10MB), validate file type (PDF, DOCX, PPTX, PNG, JPG), generate a storage path scoped to the session (wizard-uploads/sessionId/timestamp-filename), upload to the bucket via adminClient, store file metadata (storage path, original filename, size, mime type, label) in wizard_answers for step 1 inside a files array in the answers jsonb, and return a signed URL valid for 1 hour so the frontend can display a preview.

### Server: File List Route

Add a GET route that takes a sessionId and returns all uploaded file metadata with fresh signed URLs. This is used when the user returns to step 1 or when the AI analysis needs to reference uploaded content.

### Server: File Delete Route

Add a DELETE route that takes a sessionId and file path, removes the file from storage, and removes the entry from the files array in wizard_answers.

### Server: AI Integration

When the analyze-business endpoint runs, it should check if the session has uploaded documents. If files exist, the endpoint should download the file content from storage (for text-extractable formats like PDF), include a summary of the uploaded documents in the Gemini prompt alongside the URL and description, and note in the analysis response which uploaded documents were referenced.

For non-text files like images, include the filename and label in the prompt as contextual hints rather than extracting content.

### Frontend: Upload Component

StepBusinessContext already has the visual upload zone. Wire it to call the upload route on the server when files are dropped or selected. Show upload progress with a progress bar. After upload, display the file in a list with filename, size, type icon, and a remove button. Store the returned signed URL for preview thumbnails. Limit to 5 files per session. Show validation errors for oversized or wrong-type files before attempting upload. Persist the file list in the wizard state so it survives page refreshes.

### Frontend: File Display

When loading a saved wizard session, fetch the file list from the server to get fresh signed URLs. Display uploaded files in step 1 and reference them in the step 4 executive summary (brief header area showing attached documents count).

### Auth and Scoping

Guest users can upload files tied to their sessionId. Authenticated users get the same behavior but their sessionId is linked to their userId via wizard_sessions. The server uses adminClient for storage operations (upload, signed URL generation, delete) because the bucket is private. RLS on wizard_answers still controls who can see the metadata.

### Storage Cleanup

Files for abandoned sessions (status abandoned and older than 30 days) should eventually be cleaned up. For now, document the cleanup pattern but do not implement automated deletion. The signed URLs expire after 1 hour, so stale URLs naturally stop working.

## File Metadata Shape in wizard_answers

The step 1 answers jsonb should include a files array where each entry has: storagePath (the bucket path), originalName (user's filename), sizeBytes (file size), mimeType (content type), label (user-selected category), uploadedAt (timestamp), and signedUrl (temporary, refreshed on load).

## Acceptance Criteria

Users can drag-and-drop or click to upload up to 5 files in wizard step 1. Files persist in Supabase Storage in a private bucket. File metadata is saved to wizard_answers and survives session reload. The AI analysis prompt includes uploaded document context when available. Files can be removed individually. Guest and authenticated users both work. No file content or signed URLs leak to unauthorized users.
