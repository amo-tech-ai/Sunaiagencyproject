## Objective

Replace all KV store (`kv_store_283466b6`) usage in edge functions with proper Supabase table queries against `wizard_sessions`, `wizard_answers`, `ai_cache`, and `ai_run_logs`. This eliminates the Figma Make workaround, enables RLS enforcement, foreign key integrity, indexing, and type safety.

## Context

The KV store was auto-generated by Figma Make as a generic document store when DDL couldn't be run. Now that proper relational tables exist (created in migrations 000000–000004), the KV store is technical debt:

| Problem | Impact |
|---------|--------|
| No RLS | All data accessible via service role key, bypassing row-level security |
| No foreign keys | Orphaned wizard data, no referential integrity |
| No indexes | Full table scans on every key lookup |
| No type safety | JSONB blobs with no schema validation |
| Duplicate model | Tables exist but aren't used; KV duplicates the same concepts |

### Current KV Key Patterns → Target Tables

| KV Key Pattern | Target Table | Notes |
|---|---|---|
| `wizard:session:{id}` | `wizard_sessions` | Full session state (current_step, org_id, user_id, status) |
| `wizard:answer:{id}:step{n}` | `wizard_answers` | Per-step answers (session_id FK, step_number, answers jsonb) |
| `wizard:analysis:{id}` | `wizard_answers` | Store as step 1 answer data or new `ai_results` jsonb column |
| `wizard:diagnostics:{id}` | `wizard_answers` | Store as step 2 answer data |
| `wizard:recommendations:{id}` | `wizard_answers` | Store as step 3 answer data |
| `wizard:readiness:{id}` | `wizard_answers` | Store as step 4 answer data |
| `wizard:roadmap:{id}` | `wizard_answers` | Store as step 5 answer data |
| `ai:cache:{sha256}` | `ai_cache` | Has `input_hash` text PK, `response` jsonb, `expires_at` timestamptz |
| `ai:log:{timestamp}-{rand}` | `ai_run_logs` | Has `prompt_type`, `model`, `tokens_used`, `duration_ms`, `session_id` FK |

## Scope

### Phase 1: Create Supabase Client Helper

**File:** `supabase/functions/server/db.tsx` (new)

Replace the KV store import with a typed Supabase client that respects the caller's auth context.

```typescript
// db.tsx — Supabase client factory for edge functions
import { createClient } from "jsr:@supabase/supabase-js@2.49.8";

/** Service-role client for server-side operations (admin, caching) */
export const adminClient = () =>
  createClient(
    Deno.env.get("SUPABASE_URL")!,
    Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!
  );

/** Per-request client scoped to the caller's JWT (respects RLS) */
export const userClient = (authHeader: string | undefined) =>
  createClient(
    Deno.env.get("SUPABASE_URL")!,
    Deno.env.get("SUPABASE_ANON_KEY")!,
    {
      global: {
        headers: authHeader ? { Authorization: authHeader } : {},
      },
    }
  );
```

**Design decisions:**
- `adminClient` is used only for `ai_cache` and `ai_run_logs` (server-side, no user context)
- `userClient` is used for `wizard_sessions` and `wizard_answers` (RLS-scoped to the user's orgs)

### Phase 2: Refactor `wizard-routes.tsx`

**File:** `supabase/functions/server/wizard-routes.tsx`

Replace all `kv.set/get/mget/getByPrefix` calls with table queries.

#### 2a. POST /wizard/save

**Before (KV):**
```typescript
await kv.set(`wizard:session:${sid}`, { ...fullState, userId, updatedAt });
await kv.set(`wizard:answer:${sid}:step${step}`, { step, data, updatedAt });
```

**After (tables):**
```typescript
import { userClient } from "./db.tsx";

// Full state save → upsert wizard_sessions
const db = userClient(c.req.header("Authorization"));
const { error } = await db.from("wizard_sessions").upsert({
  id: sid,
  org_id: orgId,          // derive from user's team_members
  user_id: userId,
  current_step: fullState.currentStep || 1,
  status: fullState.status || "in_progress",
  context_snapshot: fullState,  // store full state as jsonb
  updated_at: new Date().toISOString(),
});

// Per-step save → upsert wizard_answers
const { error: ansError } = await db.from("wizard_answers").upsert(
  {
    session_id: sid,
    step_number: step,
    answers: data,
    updated_at: new Date().toISOString(),
  },
  { onConflict: "session_id,step_number" }  // unique constraint
);
```

**Key changes:**
- Get `org_id` from `get_user_org_ids()` or look up from `team_members` for the authenticated user
- For anonymous users, create a session with `org_id = NULL` and `user_id = NULL` (the anon-accessible RLS policy on `wizard_sessions` allows this)
- Use `upsert` with the unique constraint on `(session_id, step_number)` for `wizard_answers`

#### 2b. GET /wizard/:sessionId

**Before (KV):**
```typescript
const session = await kv.get(`wizard:session:${sessionId}`);
const answers = await kv.mget(stepKeys);
```

**After (tables):**
```typescript
const db = userClient(c.req.header("Authorization"));

const { data: session } = await db
  .from("wizard_sessions")
  .select("*")
  .eq("id", sessionId)
  .single();

const { data: answers } = await db
  .from("wizard_answers")
  .select("*")
  .eq("session_id", sessionId)
  .order("step_number");
```

#### 2c. GET /wizard/list/:userId

**Before (KV):**
```typescript
const sessions = await kv.getByPrefix(`wizard:session:`);
const userSessions = sessions.filter((s) => s.userId === userId);
```

**After (tables):**
```typescript
const db = userClient(c.req.header("Authorization"));

const { data: sessions } = await db
  .from("wizard_sessions")
  .select("id, current_step, status, created_at, updated_at")
  .eq("user_id", userId)
  .order("updated_at", { ascending: false });
```

RLS automatically filters to sessions the user can see (via org membership).

### Phase 3: Refactor `gemini.tsx`

**File:** `supabase/functions/server/gemini.tsx`

Replace KV-based caching and logging with `ai_cache` and `ai_run_logs` tables.

#### 3a. Cache: getCachedResult / setCachedResult

**Before (KV):**
```typescript
const cached = await kv.get(`ai:cache:${hash}`);
await kv.set(`ai:cache:${hash}`, { response, model, createdAt });
```

**After (tables):**
```typescript
import { adminClient } from "./db.tsx";

async function getCachedResult(hash: string) {
  const db = adminClient();
  const { data } = await db
    .from("ai_cache")
    .select("response")
    .eq("input_hash", hash)
    .gt("expires_at", new Date().toISOString())
    .maybeSingle();
  return data?.response ?? null;
}

async function setCachedResult(
  hash: string,
  response: unknown,
  model: string,
  tokensUsed: number,
  ttlHours: number = 24
) {
  const db = adminClient();
  await db.from("ai_cache").upsert({
    input_hash: hash,
    response,
    model,
    tokens_used: tokensUsed,
    expires_at: new Date(Date.now() + ttlHours * 3600000).toISOString(),
  });
}
```

**Note:** `ai_cache` uses `input_hash` as its primary key, so upserts naturally deduplicate. The `expires_at` column replaces the TTL-by-convention approach (KV had no expiry mechanism at all). Use `adminClient` because cache is server-internal, not user-scoped.

#### 3b. Logging: logAIRun

**Before (KV):**
```typescript
await kv.set(`ai:log:${Date.now()}-${rand}`, { promptType, model, tokens, duration, sessionId });
```

**After (tables):**
```typescript
async function logAIRun(params: {
  sessionId?: string;
  orgId?: string;
  promptType: string;
  model: string;
  tokensUsed: number;
  durationMs: number;
  success: boolean;
  errorMessage?: string;
}) {
  const db = adminClient();
  await db.from("ai_run_logs").insert({
    session_id: params.sessionId || null,
    org_id: params.orgId || null,
    prompt_type: params.promptType,
    model: params.model,
    tokens_used: params.tokensUsed,
    duration_ms: params.durationMs,
    success: params.success,
    error_message: params.errorMessage || null,
  });
}
```

### Phase 4: Refactor `ai-routes.tsx`

**File:** `supabase/functions/server/ai-routes.tsx`

Replace KV result persistence with `wizard_answers` writes. Each AI endpoint currently stores its result as a separate KV key; instead, store as the `ai_results` field on the corresponding `wizard_answers` row.

#### 4a. Per-endpoint result storage

**Before (KV):**
```typescript
// analyze-business
await kv.set(`wizard:analysis:${sessionId}`, result);
// industry-diagnostics
await kv.set(`wizard:diagnostics:${sessionId}`, result);
// system-recommendations
await kv.set(`wizard:recommendations:${sessionId}`, result);
// readiness-score
await kv.set(`wizard:readiness:${sessionId}`, result);
// generate-roadmap
await kv.set(`wizard:roadmap:${sessionId}`, result);
```

**After (tables):**
```typescript
import { adminClient } from "./db.tsx";

// Each AI result maps to a wizard step:
// analyze-business     → step 1 ai_results
// industry-diagnostics → step 2 ai_results
// system-recommendations → step 3 ai_results
// readiness-score      → step 4 ai_results
// generate-roadmap     → step 5 ai_results

async function saveAIResult(sessionId: string, stepNumber: number, result: unknown) {
  const db = adminClient();
  // Upsert the wizard_answers row, merging AI results alongside user answers
  await db.from("wizard_answers").upsert(
    {
      session_id: sessionId,
      step_number: stepNumber,
      ai_results: result,
      updated_at: new Date().toISOString(),
    },
    { onConflict: "session_id,step_number", ignoreDuplicates: false }
  );
}

// Usage in each endpoint:
// POST /analyze-business
await saveAIResult(sessionId, 1, analysisResult);
// POST /industry-diagnostics
await saveAIResult(sessionId, 2, diagnosticsResult);
// POST /system-recommendations
await saveAIResult(sessionId, 3, recommendationsResult);
// POST /readiness-score
await saveAIResult(sessionId, 4, readinessResult);
// POST /generate-roadmap
await saveAIResult(sessionId, 5, roadmapResult);
```

#### 4b. readiness-score data aggregation

**Before (KV):**
```typescript
const [session, analysis, diagnostics] = await kv.mget([
  `wizard:session:${sessionId}`,
  `wizard:analysis:${sessionId}`,
  `wizard:diagnostics:${sessionId}`,
]);
```

**After (tables):**
```typescript
const db = adminClient();
const { data: answers } = await db
  .from("wizard_answers")
  .select("step_number, answers, ai_results")
  .eq("session_id", sessionId)
  .in("step_number", [1, 2, 3])
  .order("step_number");

const step1 = answers?.find((a) => a.step_number === 1);
const step2 = answers?.find((a) => a.step_number === 2);
const analysis = step1?.ai_results;
const diagnostics = step2?.ai_results;
```

## Failure Handling

- If RLS blocks wizard saves for anonymous users, add a policy: `CREATE POLICY wizard_anon_insert ON wizard_sessions FOR INSERT TO anon WITH CHECK (user_id IS NULL)`
- If `wizard_answers` unique constraint doesn't exist on `(session_id, step_number)`, create it in the migration
- If `ai_results` column causes conflicts with existing data, use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`
- If edge function deploy fails, check Deno import paths — `db.tsx` must use the same Supabase SDK version as `kv_store.tsx`

## Future Cleanup (Out of Scope)

- Drop `kv_store_283466b6` table after confirming no data loss
- Add database-level expiry for `ai_cache` (pg_cron job to delete expired rows)
- Add `context_snapshots` table integration for full wizard state history
- Implement Supabase Realtime subscriptions for live wizard collaboration
